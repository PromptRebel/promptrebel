Lab Notes

Ich habe fast alles hingeschmissen – und dann lief es.

Wie ein lokaler KI-Podcast-Agent auf schwacher Hardware entstanden ist. Ohne GPU. Ohne Budget. Ohne API. Mit echtem Frust, echten Fehlern, RAM-Grenzen, Architekturwechseln und schließlich einem kompletten End-to-End-Flow: iPhone → WebUI → Surface → Ollama → XTTS → WAV.

> hardware: surface go 2 > stack: python + ollama + xtts_v2 + flask + tailscale > budget: 0 € > cloud: none > runtime model: local-first > mood: frustrated → running # promptrebel = nachbaubar vor perfekt

Ich bin kein Entwickler. Ich bin kein ITler. Und irgendwann saß ich vor einem Terminal, das mir Dinge um die Ohren geworfen hat, die ich nicht einmal sauber aussprechen konnte.

Transformers. Torch. Dependencies. Python-Pfade. Execution Policies. Webhooks. WAV-Parameter. Memory Errors. Vieles davon habe ich am Anfang nicht verstanden. Aber ich habe jeden Stolperstein dokumentiert, weil genau daraus die eigentliche Hilfe für andere entsteht.

ausgangslage

Was überhaupt gebaut werden sollte

Das Ziel war von Anfang an nicht bloß eine einzelne Audio-Datei, sondern ein kompletter lokaler Workflow: Thema oder später URL eingeben → Text lokal erzeugen → Stimme lokal erzeugen → fertige Podcast-Datei erhalten.

Wichtig dabei: keine laufenden API-Kosten, kein Cloud-Zwang, kein fertiger Baukasten, den man nur anklickt, sondern ein eigenes System, das auf echter schwacher Hardware laufen kann und mit einer guten Anleitung von anderen reproduzierbar ist.

build cases

Jeder relevante Widerstand als rekonstruierter Fall

Im Folgenden werden die wichtigsten Bruchstellen nicht als lose Fehlerliste gezeigt, sondern als Fallstudien. Genau das macht aus einem chaotischen Build ein Modell, von dem andere wirklich profitieren können.

Build Case 01

XTTS / Coqui Paketkonflikt: der erste echte Gegenstoß

Widerstand
XTTS ließ sich nicht sauber im bestehenden Python-Zustand starten. Alte und neue Paketlogik kollidierten direkt.
Rohes Symptom / Fehlermeldung
ImportError: coqui-tts switched to a forked version of Coqui,
but you still have the original package installed.
Run: pip uninstall coqui → pip install coqui-tts
Debug-Autopsie
Das System war nicht defekt, sondern logisch vermischt. Ein alter Paketstand und der neue Fork versuchten gleichzeitig, die gleiche Rolle zu spielen. Solche Konflikte wirken wie „mystische Python-Probleme“, sind aber oft schlicht Environment-Vermischungen.
Entscheidung
Nicht weiter im kaputten Zustand herumflicken. Neuer Ordner. Neue venv. Sauberer Neustart statt Reparaturversuch.
Ergebnis
Der Weg war wieder kontrollierbar. Das Projekt hatte ein Fundament, statt nur einen angesammelten Paketfriedhof.
Übertragbare Lehre
Wenn Python-Pakete sich gegenseitig hochschaukeln, ist ein neues Environment oft schneller und ehrlicher als zehn weitere Korrekturversuche.
Build Case 02

Versionen, die „irgendwie“ installierbar, aber praktisch unbrauchbar waren

Widerstand
Torch, Transformers, TTS und Windows spielten nicht automatisch sauber zusammen. Formal installierte Pakete bedeuteten noch lange kein funktionierendes System.
Rohes Symptom / Fehlermeldung
ERROR: Could not find a version that satisfies the requirement coqui-tts==0.22.0
(from versions: 0.24.1, 0.24.2, 0.24.3 ...)
PyTorch was not found. Models won't be available
PyTorch >= 2.4 is required but Found 2.1.0
Debug-Autopsie
Der Engpass war keine einzelne falsche Zeile, sondern eine komplette Versionsmatrix. Gerade auf Windows und mit CPU-only Setup war relevant, Torch gezielt zu pinnen und Transformers nicht blind aktuell zu lassen.
Entscheidung
Gezielte Versionen statt „letzte verfügbare“. Python 3.11.7. Torch CPU Build 2.4.0. Später explizite Eingriffe bei Transformers.
Ergebnis
XTTS wurde überhaupt erst ansprechbar. Das war noch keine stabile Pipeline, aber ein echter Durchbruch.
Übertragbare Lehre
In lokalen KI-Stacks ist Versionenkontrolle Teil der Architektur. Wer das ignoriert, baut Fehler bereits vor dem ersten echten Lauf ein.
Build Case 03

Der Wendepunkt: zum ersten Mal sprach XTTS wirklich

Widerstand
Zwischen „es ist installiert“ und „es erzeugt wirklich eine Datei“ lag noch eine psychologisch kritische Lücke. Genau dort geben viele auf.
Rohes Symptom / Fehlermeldung
Es wirkte lange, als passiere nichts. Modell-Download, CLI, Voice-Referenz, Warten. Kein klarer Beweis, dass am Ende tatsächlich Audio entsteht.
Debug-Autopsie
Das Problem war jetzt nicht mehr reine Installation, sondern Unsicherheit. Ist der Prozess nur lang? Hängt er? Ist die Referenzstimme korrekt? Die richtige Antwort war: eine brutal kleine Minimalprobe.
Entscheidung
Ein einzelner deutscher Satz, eine lokale voice.wav, ein Ausgabeziel. Keine Pipeline, keine WebUI, keine Zusatzlogik.
Ergebnis
XTTS erzeugte erstmals lokal eine echte Audio-Datei.
=== Generierter Text ===
"Hallo, das ist mein erster selbst gebauter KI Podcast Agent."

→ output.wav erstellt
→ Prozess beendet. Keine Fehler.
Übertragbare Lehre
Vor jeder großen Pipeline braucht es einen gnadenlosen Minimaltest. Erst wenn ein einziger Satz zuverlässig spricht, lohnt sich jede weitere Automatisierung.
Build Case 04

Von Terminal zu WebUI: der Agent sollte vom iPhone triggerbar sein

Widerstand
Ein funktionierendes Terminal-Skript war gut, aber noch kein praktischer Agent. Das Ziel war, vom iPhone aus ein Thema an das Surface zu schicken.
Rohes Symptom / Fehlermeldung
Telegram und n8n führten sofort in Webhook-, HTTPS- und Architekturprobleme. Lokal laufende Dienste sind nicht automatisch von außen triggerbar.
Debug-Autopsie
Das eigentliche Problem war nicht „wie kommt Text an“, sondern die unnötige Komplexität eines Messenger- oder Webhook-Modells für ein eigentlich lokales Projekt. Die sauberere Lösung war eine Weboberfläche im selben Netzwerkraum wie das Surface – also per Tailscale.
Entscheidung
Flask-WebUI auf dem Surface. Tailscale auf Surface und iPhone. Textfeld + Button statt Bot-Setup.
Ergebnis
Das iPhone konnte die lokale Oberfläche öffnen und den Podcast-Lauf remote starten. Kein Telegram-Zwang, keine öffentliche URL, kein Cloud-Service.
Übertragbare Lehre
Nicht jede Eingabe braucht gleich Workflow-Automation oder Bots. Oft ist eine kleine lokale WebUI die robustere und verständlichere Lösung.
Build Case 05

Die wirkliche Grenze hieß RAM

Widerstand
Sobald Textgenerierung, XTTS und WebUI zusammenspielten, war das eigentliche Problem nicht mehr Logik, sondern Speicherdruck.
Rohes Symptom / Fehlermeldung
RuntimeError: not enough memory
Zugriff auf Arbeitsspeicher konnte nicht durchgeführt werden
Beobachtung parallel im Task-Manager: Ollama hielt nach Generierung noch rund 4.2 GB RAM belegt.
Debug-Autopsie
XTTS allein war nicht das Problem. Kritisch wurde die Kombination aus Windows, Hintergrundprozessen, Ollama im Speicher, Python und Torch/XTTS-Peaks. Genau diese Lastspitze brachte das Surface an die Grenze.
Entscheidung
Textgenerierung und Sprachsynthese wurden in getrennte Python-Dateien aufgeteilt. Zusätzlich wurde Ollama nach der Textgenerierung aktiv beendet, um RAM freizugeben.
Ergebnis
Die Pipeline war nicht mehr monolithisch, sondern zweistufig. Das reduzierte die Peak-Last massiv und machte stabile Läufe realistisch.
Übertragbare Lehre
Auf schwacher Hardware ist Prozessarchitektur oft wichtiger als reine Modellwahl. Lastspitzen entkoppeln schlägt häufig bloßes Hoffen auf „es wird schon laufen“.
Build Case 06

Segmentierung statt Alles-auf-einmal

Widerstand
Ein kompletter Podcast-Text am Stück war für XTTS zu schwer. Der Gesamttext war nicht das Problem – die zu große Verarbeitungseinheit war es.
Rohes Symptom / Fehlermeldung
Crashs oder fehlerhafte Läufe beim Versuch, zu viel Text in einem Rutsch an XTTS zu geben.
Debug-Autopsie
Lange Podcasts müssen nicht als ein TTS-Block gedacht werden. Besser: großer Gesamttext, aber kleine Segmente für die Sprachsynthese. Genau dieses Denken machte den späteren stabilen Build überhaupt erst möglich.
Entscheidung
TXT lesen → in Sätze zerlegen → jeden Satz einzeln vertonen → WAVs zusammenfügen. Dazu kamen Zwischenprüfungen, Debug-Ausgaben und harte Limits, die später schrittweise erhöht wurden.
Ergebnis
Die Sprachsynthese wurde kontrollierbar. Statt Alles-oder-nichts-Läufen entstand ein modularer Build, der auf dem Surface wirklich tragfähig war.
Übertragbare Lehre
Lange Inhalte sollten auf schwacher Hardware nicht als großer TTS-Brocken gedacht werden, sondern als Liste kleiner, sauber zusammensetzbarer Segmente.
Build Case 07

Piper als Zwischenweg – und warum XTTS trotzdem blieb

Widerstand
XTTS war hochwertig, aber fragil. Piper war leichter, aber nicht das eigentliche Ziel. Es musste geklärt werden, ob Stabilität oder Stimmqualität wichtiger ist.
Rohes Symptom / Fehlermeldung
Piper war zunächst leichter installierbar gedacht, brachte aber eigene kleinere Paketprobleme mit. XTTS hatte dafür weiter den besseren Qualitäts-Charakter.
Debug-Autopsie
Piper war in Wahrheit kein vollwertiger Ersatz für die XTTS-Vision, sondern eher ein Stabilitätsanker. Die größere Erkenntnis war: die Segment- und Prozesslogik war der eigentliche Schlüssel, nicht der reine Modellwechsel.
Entscheidung
XTTS blieb Hauptpfad. Piper diente eher als technischer Vergleich und Fallback-Idee. Die eigentliche Optimierung konzentrierte sich wieder auf RAM-Freigabe, Segmentierung und Timing.
Ergebnis
XTTS lief schließlich sauber genug durch, sodass das System als lokaler Podcast-Agent funktionierte, ohne auf das hochwertigere Zielmodell verzichten zu müssen.
Übertragbare Lehre
Ein leichteres Modell kann als Fallback sinnvoll sein. Aber manchmal liegt die eigentliche Lösung nicht im Modellwechsel, sondern in der Systemarchitektur drumherum.
Build Case 08

Stabilität beweisen statt nur behaupten

Widerstand
Ein einziger erfolgreicher Lauf ist noch kein belastbarer Zustand. Die Frage war: läuft das System wiederholt, sauber und messbar?
Rohes Symptom / Fehlermeldung
Keine Fehlermeldung mehr im klassischen Sinn – eher die Gefahr, vorschnell zu früh zu viel zu wollen, etwa direkt 20–30 Minuten Podcast auf Hardware, die gerade erst kurze Monologe sauber schafft.
Debug-Autopsie
Ein funktionierender Lauf beweist nur Möglichkeit. Mehrere erfolgreiche Läufe mit Stoppuhr beweisen Reproduzierbarkeit. Genau deshalb wurde die Ausgabe schrittweise erhöht und jeweils mehrfach getestet.
Entscheidung
Limits nicht sofort komplett aufheben, sondern moderat erhöhen. Beispiel: erst 4 Segmente, dann 8 Segmente und 140 Zeichen, mehrfach fehlerfrei. Erst danach wäre mehr Länge sinnvoll.
Ergebnis
Der Agent lief mehrfach komplett durch. Damit war nicht mehr nur ein Demo-Moment da, sondern ein funktionierender Build-Status.
Übertragbare Lehre
Stabilität entsteht nicht durch Hoffnung, sondern durch kontrollierte Belastung. Erst klein stabil, dann größer.

│

├─ podcast.py                # frühe Zwischenstufe / Testphase
├─ voice.wav                 # lokale Referenzstimme für XTTS
├─ status.json               # Statusdatei für WebUI
├─ segments\                # temporäre WAV-Segmente
├─ output.wav                # frühe Testdatei
├─ podcast_YYYYMMDD_HHMMSS.txt
└─ podcast_YYYYMMDD_HHMMSS.wav

Welche Datei macht was?

  • generate_text.py erzeugt den Text mit Ollama, speichert TXT, beendet Ollama, startet die Vertonung.
  • speak_text.py liest die TXT, segmentiert, vertont Satz für Satz mit XTTS und fügt die WAVs zusammen.
  • webui.py stellt die lokale Oberfläche bereit, startet den Build und zeigt Status / Fortschritt / Ergebnis.
  • voice.wav ist die XTTS-Referenzstimme.
  • status.json ist die Kommunikation zwischen Build-Prozess und Oberfläche.

Wirklich relevante Stolperstellen aus der Praxis
  • Mehrfach war der Fehler schlicht: Internet war nicht aktiv.
  • Dateien wurden unter Windows versehentlich ohne echte .py-Endung gespeichert.
  • PowerShell-Execution-Policy blockierte anfangs npm/npx und andere Skripte.
  • XTTS wirkte manchmal kaputt, obwohl in Wahrheit nur der Speicherpeak zu hoch war.
  • Telegram/n8n war nicht der saubere Weg für diesen lokalen Workflow.
  • Wenn das Surface in echten Standby geht, ist der Dienst weg. Bildschirm aus ist okay, Standby nicht.
  • Browser offen halten ist nicht nötig – der laufende Python-/Flask-Prozess ist entscheidend.
workflow heute

Was der fertige Workflow heute wirklich macht

# 1. iPhone öffnet die WebUI über Tailscale
# 2. Thema wird im Browser eingegeben
# 3. webui.py startet generate_text.py
# 4. generate_text.py ruft Ollama auf
# 5. Ollama generiert den Podcast-Text
# 6. Der Text wird als podcast_*.txt gespeichert
# 7. Ollama wird beendet, um RAM freizugeben
# 8. generate_text.py startet speak_text.py
# 9. speak_text.py liest die TXT-Datei
# 10. Der Text wird in Segmente zerlegt
# 11. XTTS vertont jedes Segment einzeln
# 12. Alle Segmente werden zu einer finalen WAV zusammengefügt
# 13. status.json wird laufend aktualisiert
# 14. Die WebUI zeigt: startet / läuft / Fortschritt / fertig
learnings

Was davon übertragbar ist

Was wirklich funktioniert hat

  • saubere neue Umgebung statt Reparatur im Altzustand
  • Minimaltests vor jeder größeren Pipeline
  • Trennung von Textgenerierung und Sprachsynthese
  • Ollama nach Nutzung beenden
  • segmentweise XTTS-Verarbeitung
  • lokale WebUI statt vorschnell Bots / Workflows
  • schrittweises Hochfahren statt Vollgas von Anfang an

Was andere daraus mitnehmen können

  • schwache Hardware heißt nicht automatisch „unmöglich“
  • viele „große“ Fehler sind am Ende konkrete, kleine Dinge
  • Frust ist oft ein Build-Abschnitt, kein Endpunkt
  • funktionierend schlägt perfekt, vor allem in frühen Phasen
  • geteilte Stolperstellen sind oft wertvoller als perfekte Ergebnisse

Genau deshalb passt dieses Projekt zu PromptRebel. Nicht weil es elegant angefangen hat. Sondern weil es dokumentiert, ehrlich und reproduzierbar geworden ist.
