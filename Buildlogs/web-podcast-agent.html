Der Podcast-Agent auf dem Surface lief. Und sofort war die nächste Frage da: Könnte das auch jemand anderes nutzen? Ohne bei mir daheim ins Netzwerk zu müssen? Ohne Tailscale? Ohne dass mein Surface läuft?

Ich bin kein Servermensch. Ich miete keine Infrastruktur. Ich will keinen VPS, der monatlich Geld kostet und dann doch irgendwann ausfällt. Die Frage war also: Geht das auch im Browser? Und wenn ja – wie weit?

ausgangslage

Die Ausgangslage: Surface funktioniert, aber es ist nur mein Surface

Nach Build Log #01 stand fest: der Podcast-Agent funktioniert. Textgenerierung mit Ollama, Sprachsynthese mit XTTS, Flask-WebUI, Tailscale – alles lokal auf dem Surface Go 2. Das war echte Infrastruktur. Und genau das war auch das Problem.

Ein Agent, der nur läuft wenn mein Surface an ist, mein Flask-Server oben ist und Tailscale verbunden ist, ist kein Tool für andere. Das ist ein Demo für mich selbst. Für PromptRebel als öffentliche GitHub-Page brauchte es etwas anderes.

„Wäre natürlich klasse, wenn ich das auch auf meiner PromptRebel GitHub Page irgendwie etablieren könnte. Also funktionsfähig etablieren."

Die entscheidende Erkenntnis zu GitHub Pages
GitHub Pages ist statisches Hosting. Kein Python, kein Node, keine Hintergrundprozesse. Was gut geht: Eingabefelder, Buttons, clientseitige Logik, kleine Modelle über WebGPU/WASM, Ergebnisse im Browser anzeigen oder als Download erzeugen. Was nicht geht: Ollama starten, XTTS serverseitig rendern, Jobs im Hintergrund queueen, API-Keys verstecken. Das ist kein Limit. Das ist die Architekturvorgabe.
architekturentscheidung

Drei Wege – und warum nur einer übrig blieb

Die erste echte Entscheidung war keine technische, sondern eine philosophische: Welche Architektur passt zu PromptRebel?

Option A

Alles im Browser

  • GitHub Pages als Hülle
  • Browser macht die Arbeit
  • WebGPU / Transformers.js
  • Kein Server, keine Kosten
Option B

GitHub Pages + Backend

  • Frontend öffentlich
  • Inferenz auf externem Server
  • VPS / Modal / Replicate
  • Kosten, Wartung, Abhängigkeit
Option C

GitHub Pages + lokaler Companion

  • Surface als lokaler API-Endpunkt
  • CORS-Probleme vorprogrammiert
  • Nur wenn mein Surface läuft
  • Nicht für andere nutzbar

Die Entscheidung war schnell klar. Option B kostet Geld und schafft Abhängigkeiten. Option C ist genau das, was schon mit Tailscale existiert – also kein Fortschritt. Option A war die einzige, die zur PromptRebel-Idee passt: transparent, reproduzierbar, ohne Server, ohne Budget, ohne Blackbox.

Was im Browser geht

  • Eingabefelder, Buttons, Player, UI
  • kleine LLMs über Transformers.js / WebGPU
  • Browser-eigene Sprachsynthese (Web Speech API)
  • echtes KI-TTS über Transformers.js (experimentell)
  • WAV-Export aus Modell-Output
  • Modell-Download + lokaler Cache

Was nicht geht (ohne Backend)

  • Ollama / Python / XTTS starten
  • Jobs im Hintergrund verarbeiten
  • API-Keys verstecken
  • große Audiodateien persistent speichern
  • hochwertige lange Podcast-Produktion für alle Geräte
v1 vs v2 – zwei ansätze

V1 und V2: nicht zwei Versionen derselben Sache

Schon früh war klar: V1 und V2 sind keine Iterationen desselben Konzepts. Sie sind zwei grundlegend verschiedene technische Ansätze. Das ist keine Kleinigkeit – das ist der eigentliche Architekturkern.

V1 – Browser-TTS

  • Kleiner LLM im Browser generiert Text
  • Web Speech API liest vor (speechSynthesis.speak())
  • Keine Audiodatei, nur Vorlesefunktion
  • Kein KI-Modell für Stimme nötig
  • Stimme = Betriebssystem-Stimme des Nutzers
  • Download: nicht zuverlässig möglich

V2 – KI-TTS im Browser

  • Kleiner LLM im Browser generiert Text
  • Transformers.js lädt echtes TTS-Modell
  • Modell erzeugt rohe Audiodaten
  • WAV-Download aus Blob möglich
  • Stimme konsistenter, KI-generiert
  • Deutlich schwerer, RAM-intensiver
Der wichtige Unterschied bei Downloads
V1 spricht. V2 erzeugt Audiodaten. Nur letzteres ist wirklich download-freundlich. Die Web Speech API gibt dir Kontrolle über die Vorlesefunktion – aber keine fertige Audiodatei. Wenn das TTS-Modell in V2 selbst die Audiodaten liefert, kann man daraus im Browser einen Blob bauen und als WAV anbieten. Das ist kein Detail, das ist der Unterschied zwischen „Vorlesen-Button" und „Download-Button".
build cases

Build Case 01

GitHub Pages als Host: Grenze verstehen, bevor man dagegen baut

Widerstand
Die Idee war: Podcast-Agent auf der GitHub-Page. Die Frage dahinter: Kann GitHub Pages das überhaupt?
Rohes Symptom
Keine Fehlermeldung – aber die Vorstellung war falsch. GitHub Pages kann kein Python, kein Ollama, kein XTTS. Es ist statisches Hosting. Wer das nicht weiß, baut gegen eine Wand.
Debug-Autopsie
Das eigentliche Missverständnis war nicht die Technik, sondern die Annahme, GitHub Pages wäre ein Server. Es ist eine Dateiauslieferung. Die Schlussfolgerung: Alles, was nicht in einer statischen Datei oder im Browser passiert, geht nicht.
Entscheidung
Kein Server. Kein Backend. Alles läuft im Browser des Nutzers – und das ist kein Kompromiss, sondern die PromptRebel-Idee in ihrer reinsten Form.
Ergebnis
Klare Architektur. GitHub Pages als Hülle. Browser als Recheneinheit. Kein Widerspruch mehr.
Übertragbare Lehre
Grenzen verstehen ist keine Niederlage. Wer weiß, was eine Plattform nicht kann, trifft bessere Architekturentscheidungen als wer es einfach versucht.
Build Case 02

Apple blockiert WebGPU – oder doch nicht?

Widerstand
Der WebGPU-Chatbot auf PromptRebel lief auf dem Surface und dem PC, aber nicht auf dem iPhone. Die Annahme: Apple blockt das grundsätzlich.
Rohes Symptom
Kein Laden des Chatbots im iPhone-Browser. Kein Fehler, einfach kein Start.
Debug-Autopsie
Die Annahme „Apple blockiert WebGPU generell" war zum Zeitpunkt des Builds bereits veraltet. WebKit hat WebGPU-Unterstützung für Safari auf macOS, iOS, iPadOS und visionOS angekündigt und ausgerollt. Das eigentliche Problem lag wahrscheinlich an iOS-/Safari-Version, Geräte-Limit oder Modell-Kompatibilität – nicht an einem generellen Apple-Block.
Entscheidung
iPhone-Kompatibilität als bekannte Einschränkung offen kommunizieren, nicht als Architekturproblem behandeln. Der Nutzer auf mobilen Geräten bekommt einen klaren Hinweis.
Ergebnis
Kein Rabbit Hole um Apple-Workarounds. Klare Nutzerkommunikation statt technischer Vernebelung.
Übertragbare Lehre
„Das Ding blockiert das" ist selten die vollständige Wahrheit. Bevor man eine Architekturentscheidung auf einer Annahme aufbaut, lohnt sich ein kurzer Gegencheck.
Build Case 03

V1 Download-Button: die Brücke, die es nicht gibt

Widerstand
V1 sollte vorlesen und einen Download-Button haben. Klingt einfach. War es nicht.
Rohes Symptom
Kein Crash, kein Fehler – nur die Erkenntnis, dass der geplante Download-Button technisch nicht sauber existiert. Die Web Speech API gibt dir keine fertige Audiodatei.
Debug-Autopsie
speechSynthesis.speak() steuert den Browser-Vorleseprozess. Aber zwischen dieser API und „hier ist dein WAV-Download" gibt es keinen standardisierten, browserübergreifenden Pfad. MediaRecorder kann Audio-Streams aufnehmen – aber nicht direkt den speechSynthesis-Stream, zumindest nicht zuverlässig über alle Browser hinweg.
Entscheidung
V1 ohne Download planen. Vorlesen-Button, Stop/Pause/Fortsetzen – das ist realistisch. Download-Button erst bei V2, wo das TTS-Modell selbst die Audiodaten liefert und man daraus einen Blob bauen kann.
Ergebnis
Sauberere V1-Architektur. Kein halbfertiger Download-Hack. V2 bekommt den Download, weil V2 ihn verdient – weil das Modell selbst die Daten liefert.
Übertragbare Lehre
Nicht jede Funktion, die sich einfach anfühlt, ist einfach zu bauen. Eine Brücke zwischen zwei APIs zu wollen heißt nicht, dass diese Brücke existiert.
Build Case 04

HuggingFace Xet Bridge – der Infrastruktur-Crash, der fast alles gestoppt hätte

Widerstand
V2 sollte ein echtes TTS-Modell im Browser laden. Stattdessen: Ladefehler, keine Audioerzeugung, keine klare Fehlermeldung. Das Modell schien nicht da zu sein, obwohl die URL stimmte.
Rohes Symptom / Fehlermeldung
Failed to load resource: HuggingFace Xet Bridge ``` TypeError: Cannot read properties of undefined (reading ‘audio’) SpeakerEmbeddings: URL not found or format mismatch
Debug-Autopsie
HuggingFace hatte seine interne Infrastruktur geändert – der sogenannte Xet Bridge, eine CDN-Zwischenschicht für Modelldateien. Modelle, die früher über direkte URLs erreichbar waren, lieferten plötzlich Fehler oder leere Responses. Dazu kamen veraltete Speaker-Embedding-URLs in der Pipeline-Konfiguration: podcast-ai-tts.js referenzierte eine alte Pfadstruktur für die Sprecherreferenz-Datei, die HuggingFace nach dem Infrastrukturwechsel nicht mehr auslieferte.
Entscheidung
Nur podcast-ai-tts.js ersetzen – nicht die ganze Seite umbauen. Neue Speaker-Embeddings-URL. Harte Neuladen im Browser. V2 Voice zuerst laden, dann Audio erzeugen.
Ergebnis
=== V2 Audio Pipeline === > Speaker Embeddings: geladen ✓ > Modell: initialisiert ✓ > Text → TTS → Audio: verarbeitet ✓ > → audio.wav erzeugt > → Download-Button aktiv > Prozess beendet. Keine Fehler.
>
>
>
>
Übertragbare Lehre
>
Externe Infrastruktur ändert sich ohne Ankündigung. Wenn ein Modell plötzlich nicht mehr lädt, obwohl sich nichts am eigenen Code geändert hat, ist der erste Verdacht: URL, CDN oder Infrastrukturwechsel auf der anderen Seite. Nicht der eigene Code.

Build Case 05

Sprache gemischt, Skript abgeschnitten – und trotzdem der Durchbruch

Widerstand
V2 erzeugte Audio. Aber: die Sprache klang englisch-deutsch gemischt, und das Skript wurde nicht vollständig gesprochen. Zwei klare Probleme, direkt nach dem ersten Erfolg.
Rohes Symptom
Hörbar englische Aussprache bei deutschem Text. Audio endet nach einem Teil des Skripts. Kein Crash – aber kein vollständiges Ergebnis.
Debug-Autopsie
Zwei getrennte Probleme mit zwei getrennten Ursachen:

Sprachmix: Das verwendete Modell (speecht5_tts) ist kein deutsches Spezialmodell. Dass die Aussprache bei deutschem Input schief klingt, ist Modell-bedingt, nicht Architektur-bedingt.

Abschneiden: TTS-Modelle kommen mit zu langem Input schlecht zurecht. Das ist kein Fehler, sondern ein Längen-/Stabilitätsproblem. Lösung: Chunking – Text in kleinere Segmente aufteilen, nacheinander vertonen, zusammenfügen. Exakt dasselbe Pattern wie bei XTTS auf dem Surface in Build Log #01.
Entscheidung
Zuerst Pipeline stabilisieren, nicht Perfektion jagen. Text künstlich kürzen (300–400 Zeichen), testen ob vollständig gesprochen wird. Erst wenn kurze Texte stabil laufen: Chunking einbauen für längere Inhalte.
Ergebnis
Der eigentliche Satz danach: „Es wird eine Audio erzeugt 🥳 – die Sprache ist zwar englisch-deutsch gemischt und beinhaltet nicht das komplette Skript aber – es funktioniert technisch und architektonisch!!!!"

Kein vollständiger Podcast. Aber ein bewiesenes Konzept. Text läuft lokal im Browser. TTS läuft lokal im Browser. Audio-Player läuft. WAV-Download läuft. Die Gesamtpipeline funktioniert.
Übertragbare Lehre
Stimme ist kein Architekturproblem. Länge ist kein Architekturproblem. Beides ist eine Qualitäts- und Konfigurationsfrage, die man angehen kann, sobald die Pipeline bewiesen ist. Erst beweisen, dann optimieren.
Build Case 06

„Ist das was Besonderes?" – Die ehrlichste Frage im ganzen Build

Widerstand
NotebookLM hatte das Projekt als „etwas Großes, eine Neuheit" beschrieben. Die Gegenfrage war direkter: Ist es das wirklich? Oder redet man sich da etwas schön?
Rohes Symptom
Kein technischer Fehler. Eine ehrliche Unsicherheit darüber, was der eigentliche Wert des Projekts ist.
Debug-Autopsie
Die nüchterne Einordnung: Technisch gesehen ist ein kleiner LLM + TTS + WAV-Export im Browser bekannte Technik. Es gibt ähnliches in Varianten – Research, Demos, GitHub-Projekte.

Aber: Der eigentliche Wert liegt woanders. Zero-Barrier-Zugang (keine Anmeldung, keine API, keine Installation). Alles lokal (kein Server, keine Datenabgabe, keine Kosten). Reproduzierbarkeit: copy – paste – läuft. Didaktischer Wert: zeigen wie es funktioniert, nicht nur dass es funktioniert.

Und am wichtigsten: es wurde umgesetzt. Die meisten bleiben bei „interessant, aber zu kompliziert".
Entscheidung
Keine Revolution behaupten. Aber auch keine unnötige Selbstverkleinerung. Das Projekt ist, was es ist: ein funktionierender Local-AI-Prototyp, ein Beweis dass sowas im Browser geht, ein Baustein auf dem man weiter aufbaut.
Ergebnis
Klares Selbstbild ohne Hype und ohne falsche Bescheidenheit. Das ist keine weiche Erkenntnis – das ist Grundvoraussetzung für gute Dokumentation.
Übertragbare Lehre
„Nicht neu im globalen Sinne" und „außergewöhnlich für das, was du gerade aufbaust" sind kein Widerspruch. Wer die eigene Arbeit richtig einordnen kann, kommuniziert sie auch ehrlicher.

learnings

Was davon übertragbar ist

Was wirklich funktioniert hat

  • V1 und V2 als getrennte Ansätze begreifen, nicht als Iterationen
  • GitHub Pages-Limits früh klären statt dagegen bauen
  • Pipeline beweisen bevor man Qualität optimiert
  • Chunking als universelles Muster für TTS-Stabilität
  • HF-Infrastruktur-Änderungen als externe Variable einplanen
  • Ehrliche Einordnung ohne Hype und ohne falsche Bescheidenheit

Was andere daraus mitnehmen können

  • Kein Server ist eine Architekturentscheidung, kein Kompromiss
  • Web Speech API und KI-TTS sind zwei verschiedene Dinge
  • Download-Button klingt einfach – ist bei V1 aber strukturell schwer
  • Wenn externe Modelle plötzlich nicht laden: URL und CDN zuerst prüfen
  • „Es erzeugt Audio" ist wichtiger als „die Stimme klingt gut"

„Es wird eine Audio erzeugt 🥳 – die Sprache ist zwar englisch-deutsch gemischt und beinhaltet nicht das komplette Skript – aber es funktioniert technisch und architektonisch!!!!"

Genau das ist der Satz, der zählt. Nicht weil er perfekt klingt. Sondern weil ab diesem Moment klar ist: die Frage ist nicht mehr „geht das überhaupt?", sondern „welches Modell, welche Textlänge, wie gut kann die Stimme werden". Das ist eine viel bessere Problemklasse.

nächste schritte

Was als Nächstes kommen kann

  • deutsches TTS-Modell testen statt englisch-dominiertem speecht5
  • Chunking einbauen für vollständige Skripte ohne Abschneiden
  • V2 stabilisieren: zuverlässig kurze Segmente, bevor man die Länge erhöht
  • klarer Hinweis auf der Seite: experimentell, limitiert, lokal
  • Build Log Guide mit kopierbaren Codeboxen – aber erst wenn V2 stabil läuft
  • Reproduzierbarkeit als eigentliches Ziel: andere sollen es nachbauen können
Die richtige Reihenfolge
Erst zum Laufen bringen, dann dokumentieren. Nicht umgekehrt. Alles andere erzeugt nur Schein-Sauberkeit: hübsche Erklärung, schöne Build Logs, aber intern noch halb instabil. Pipeline stabil → Qualität stabil → Build Log Guide. In dieser Reihenfolge. Nicht früher.
